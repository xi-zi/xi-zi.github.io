House of some

这是Csome师傅提出的一种针对较高版本的IO_FILE漏洞利用技巧。

链接：Bring back the stack attack – House of some一种高版本glibc的利用思路 · Csome

该手法利用范围广。攻击提权靠栈上ROP，同时，即使_wide_data->vtable加上检测，只允许执行虚表内的函数，该手法也可以稳定getshell。

这里首先需要_IO_new_file_underflow这个函数。

我们需要走到后面这个_IO_SYSREAD。那我们就需要满足一些条件。

1.fp->_flags & _IO_EOF_SEEN(0x10) = 0

2.fp->_flags & _IO_NO_READS(0x4) = 0

3.fp->_IO_read_ptr >= fp->_IO_read_end

4.fp->_IO_buf_base != NULL

5.fp->_flags & (_IO_LINE_BUF(0x200)|_IO_UNBUFFERED(0x2)) = 0

再看一下_IO_flush_all函数

可以看出来，对于_IO_list_all，通过_chain串起来，并且会遍历链表上的每一个FILE。同时，如果满足条件，就可以触发_IO_OVERFLOW。

由于_IO_new_file_underflow内有一个_IO_switch_to_get_mode函数其中有这个分支

我们在虚表里做了手脚（比如利用偏移错位），让 _IO_OVERFLOW 实际上跳到了 _IO_new_file_underflow。

执行流进入 _IO_new_file_underflow，并且顺理成章地调用了内部函数 _IO_switch_to_get_mode。如果我们还是通过fp->_IO_write_ptr > fp->_IO_write_base来触发_IO_OVERFLOW，就会导致利用链为_IO_flush_all -> _IO_OVERFLOW -> _IO_new_file_underflow -> _IO_switch_to_get_mode -> _IO_OVERFLOW -> ...所以这里会存在无限递归。最后导致栈空间被光速耗尽，程序直接 Crash。

那如果fp->_IO_write_ptr > fp->_IO_write_base这条分支不可行。我们就只能通过后面的分支来触发_IO_OVERFLOW了。

这里由于fp->_IO_write_ptr > fp->_IO_write_base为假，所以就要导致后面的结果都为真，才会触发_IO_OVERFLOW。也就是要满足以下条件。

1._IO_vtable_offset (fp) == 0

2.fp->_mode > 0

3.fp->_wide_data->_IO_write_ptr > fp->_wide_data->_IO_write_base

那么实现任意地址写的fake file设置如下

  • _flags设置为~(2 | 0x8 | 0x800)，设置为0即可（与apple2相同）
  • vtable设置为_IO_wfile_jumps/_IO_wfile_jumps_mmap地址，使得调用_IO_wfile_overflow即可（注意此处与apple2不同的是，此处的vtable不能加偏移，否则会打乱_IO_SYSREAD的调用）
  • _wide_data->_IO_write_base设置为0，即满足*(_wide_data + 0x18) = 0（与apple2相同）
  • _wide_data->_IO_write_ptr设置为大于_wide_data->_IO_write_base，即满足*(_wide_data + 0x20) > *(_wide_data + 0x18)（注意此处不同）
  • _wide_data->_IO_buf_base设置为0，即满足*(_wide_data + 0x30) = 0（与apple2相同）
  • _wide_data->_wide_vtable设置为任意一个包含_IO_new_file_underflow，其中原生的vtable就有，设置成_IO_file_jumps-0x48即可
  • _vtable_offset设置为0
  • _IO_buf_base与_IO_buf_end设置为你需要写入的地址范围
  • _chain设置为你下一个触发的fake file地址
  • _IO_write_ptr <= _IO_write_base即可
  • _fileno设置为0，表示read(0, buf, size)
  • _mode设置为2，满足fp->_mode > 0即可
