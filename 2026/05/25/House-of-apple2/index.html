House of apple2

这里还是看下roderick师傅的文章便于理解。

链接：[原创] House of apple 一种新的glibc中IO攻击方法 (2)-Pwn-看雪安全社区｜专业技术交流与安全研究论坛

由于高版本下，vtable无法被直接伪造（有严格的范围检查）。所以我们主动去调用_wide_data结构体里的函数指针。而_wide_data及其对应的虚表没有任何安全检查机制。通过控制这个“法外之地”，我们达到目的。

我这里主要是记录下几条攻击链条。

1.利用_IO_wfile_overflow函数控制程序执行流

首先看一下_IO_wfile_overflow函数。

这里首先要通过三个检测点。

1.(f->_flags & _IO_NO_WRITES) = 0，而_IO_NO_WRITES是0x8。

2.(f->_flags & _IO_CURRENTLY_PUTTING) = 0，而_IO_CURRENTLY_PUTTING是0x800。

3(f->_wide_data->_IO_write_base) = 0。控制_wide_data->_IO_write_base = 0

所以我们_flags应该设置为~（ 0x8 | 0x800）。这样保证触发_IO_wdoallocbuf。

_IO_wdoallocbuf

1.(fp->_wide_data->_IO_buf_base) = 0。要控制_wide_data->_IO_buf_base = 0

2.(fp->_flags & _IO_UNBUFFERED) = 0，而_IO_UNBUFFERED是2。所以_flags应该设置为~2

_IO_UNBUFFERED 代表“无缓冲区模式”。所以逻辑为如果我们这里文件流不是无缓冲模式，系统就认为我们需要一个缓冲区。然后就会去调用_IO_WDOALLOCATE来为我们分配一个缓冲区。

_IO_WDOALLOCATE的一参是fp。而_IO_WDOALLOCATE位于_wide_vtable的0x68偏移处。所以如果我们把这个地方写入system函数的地址。同时控制fp=sh就可以getshell了。

现在来总结下利用链。

  • 首先把_flags设置为~（2 | 0x8 | 0x800）。如果对一参没有要求，直接控制为0就可以。如果需要获得shell。就可以设置为 sh;。这前面有两个空格。因为这样的十六进制表现为0x3b68732020。将其写入_flags的前四字节为0x68732020，刚好满足了_flags的要求，同时也能拿到shell。
  • vtable设置为_IO_wfile_jumps或_IO_wfile_jumps_mmap/_IO_wfile_jumps_maybe_mmap地址（加减偏移）。因为在 _IO_wfile_jumps 这个官方虚表中，偏移 0x18 的位置恰好就是 _IO_wfile_overflow，而虚表中的 __overflow 接口偏移固定为 0x18。所以加减偏移是为了最终能调用_IO_wfile_overflow
  • _wide_data设置为可控堆地址A，即满足*(fp + 0xa0) = A
  • _wide_data->_IO_write_base设置为0，即满足*(A + 0x18) = 0
  • _wide_data->_IO_buf_base设置为0，即满足*(A + 0x30) = 0
  • _wide_data->_wide_vtable设置为可控堆地址B，即满足*(A + 0xe0) = B
  • _wide_data->_wide_vtable->doallocate设置为地址C用于劫持RIP，即满足*(B + 0x68) = C

2.利用_IO_wfile_underflow_mmap函数控制程序执行流

首先看_IO_wfile_underflow_mmap

1.(fp->_flags & _IO_NO_READS) = 0，_IO_NO_READS是4。

2.fp->_wide_data->_IO_read_ptr >= fp->_wide_data->_IO_read_end，这说明宽字符的缓冲区已经空了，需要补充新数据（触发overflow）。

3.fp->_IO_read_ptr < fp->_IO_read_end。这样系统就会认为底层原始字节缓冲区里还有数据没处理完，然后就不会执行后续的_IO_file_underflow_mmap，而是直接往下走。

4.fp->_wide_data->_IO_buf_base == NULL，欺骗系统为还没有分配缓冲区，其实到这一步就已经可以调用_IO_wdoallocbuf了。

5.fp->_wide_data->_IO_save_base = NULL，这是为了防止执行 free() 时因为传入非法地址而崩溃。（如果有合法的可被free的地址也可以传入。一般设置为0）

利用链：

  • _flags设置为~4。其余和上面要求一样。
  • vtable设置为_IO_wfile_jumps_mmap地址（加减偏移），使其能成功调用_IO_wfile_underflow_mmap
  • _IO_read_ptr < _IO_read_end，即满足*(fp + 8) < *(fp + 0x10)
  • _wide_data设置为可控堆地址A，即满足*(fp + 0xa0) = A
  • _wide_data->_IO_read_ptr >= _wide_data->_IO_read_end，即满足*A >= *(A + 8)
  • _wide_data->_IO_buf_base设置为0，即满足*(A + 0x30) = 0
  • _wide_data->_IO_save_base设置为0或者合法的可被free的地址，即满足*(A + 0x40) = 0
  • _wide_data->_wide_vtable设置为可控堆地址B，即满足*(A + 0xe0) = B
  • _wide_data->_wide_vtable->doallocate设置为地址C用于劫持RIP，即满足*(B + 0x68) = C

3.利用_IO_wdefault_xsgetn函数控制程序执行流

这条链执行的条件是调用到_IO_wdefault_xsgetn时rdx寄存器，也就是第三个参数不为0。如果不满足这个条件，可选用其他链。

_IO_wdefault_xsgetn

_IO_wdefault_xsgetn -> __wunderflow

当程序试图读取宽字符时（设置 _mode > 0，告诉系统这是一个宽字符流），会进入 _IO_wdefault_xsgetn。

这里more就是n，表示的是“还需要读多少个字节”。也放在了三参的位置。所以正好就是rdx的位置。

然后最后有一个检测，所以我们不能让more = 0，不然就直接break了。

count是计算当前宽字符缓冲区里还有多少未读的有效数据。如果count > 0，会去与more进行比较。如果是count大，就说明缓冲区内的数据比我需要的多，那我就只会读取我需要的量。

这有一个优化。如果count的值>20，会去调用__wmempcpy，速度快。如果count的值<=20，就直接写一个while循环，一个一个拷贝到目标缓冲区。

如果count小，那么也就是more - count > 0，就会进__wunderflow (fp)。即去底层文件/操作系统去读。

最后返回n - more，表示实际成功读取的字符。

所以我们这里去满足fp->_wide_data->_IO_read_end - fp->_wide_data->_IO_read_ptr = 0。这样count = 0，然后就不会触发拷贝的环节。直接走到了最后if的语句。即if (more == 0 || __wunderflow (fp) == WEOF)。由于我们的more肯定不会为0。所以会进__wunderflow。

__wunderflow

1.fp->_mode > 0，为了通过前两个if。如果 _mode == 0：程序会调用 _IO_fwide (fp, 1) 试图初始化宽字符状态。但因为我们是伪造的结构体，很多内部状态不对，调用这个复杂的初始化函数极有可能引发崩溃。

2._IO_int_put_mode (fp)是宏定义。

所以我们要满足_flags为0x800。因为_IO_CURRENTLY_PUTTING的值就是0x800。

这里是为了制造身份冲突。我们是因为数据不够所以进入了__wunderflow来读数据的。但是我们设置了_flag为0x800，这就告诉系统，现在的状态是写。所以这就引起读写冲突。需要先做模式切换（清理写模式）。于是会立刻调用_IO_switch_to_wget_mode。

1.fp->_wide_data->_IO_write_ptr < fp->_wide_data->_IO_write_base。在切换到读模式之前，系统必须确保之前“正在写”的数据没有丢失。所以这里是假装有残留数据。表示写缓冲区里还有没被保存的残留数据。

之后就会调用_IO_WOVERFLOW来刷新这些数据。

调用_IO_WOVERFLOW(fp)时，实际上被替换成了WJUMP1，然后FUNC，被替换__overflow，THIS被替换为fp。_IO_WIDE_JUMPS_FUNC是核心。它提取了fp结构体中的_wide_data成员，然后进一步指向了_wide_vtable。所以最后经过一些列宏展开后，实际上执行了：

  • _flags设置为0x800。（如果要getshell的话，可以改成1;sh）
  • vtable设置为_IO_wstrn_jumps/_IO_wmem_jumps/_IO_wstr_jumps地址（加减偏移），使其能成功调用_IO_wdefault_xsgetn即可。
  • _mode设置为大于0，即满足*(fp + 0xc0) > 0
  • _wide_data设置为可控堆地址A，即满足*(fp + 0xa0) = A
  • _wide_data->_IO_read_end == _wide_data->_IO_read_ptr设置为0，即满足*(A + 8) = *A
  • _wide_data->_IO_write_ptr > _wide_data->_IO_write_base，即满足*(A + 0x20) > *(A + 0x18)
  • _wide_data->_wide_vtable设置为可控堆地址B，即满足*(A + 0xe0) = B
  • _wide_data->_wide_vtable->overflow设置为地址C用于劫持RIP，即满足*(B + 0x18) = C

[巅峰极客 2022]happy note

链接：[巅峰极客 2022]happy note - NSSCTF

环境是ubuntu22.04的，但是不知道为什么本地做的时候一直patch不上依赖，加上题解区的师傅们用的glibc也都存在差异。所以只打通了本地。不过也只是用来练习house of apple2这一手法。

菜单题。然后这里的delete函数里不存在漏洞。不过特意给了个uaf的，而且只能用一次。

这里关键点是add函数 。存在两种形式去申请chunk。一种是malloc，另一种是calloc。而且只给了两次机会去申请malloc。

两者区别在于calloc申请的同时会初始化chunk为0。

所以思路是先通过塞满tcache bins，从而释放chunk到unsorted bin来获取libc基地址和堆地址。

然后由于释放两个物理地址相邻的chunk会在unsorted bin中导致合并，从而无法泄露堆地址。所以这里中间用了chunk8去分割 。

得到libc基地址和堆地址以后就可以进行apple2的利用了。触发点在delete函数。

当我们释放两次相同idx的chunk时，会触发exit(0)。从而执行我们伪造的IO流。

1._IO_wfile_overflow

2._IO_wfile_underflow_mmap

3._IO_wdefault_xsgetn

这第三个打法要解释一下。这里为什么要把_flags改为1;sh而不是上面两种。是因为在第三条利用链中需要满足_flags & 0x800。所以我们这样修改以后刚好能通过。而如果改成 ;sh会导致system接收到的第一条指令就是空格，不能另起一个shell。所以这里传入一个1让system能正常运行。1是识别不了的，但是第二个system(“sh”)就可以getshell了。
