2.Smallbox

简单的程序。就是让我们往里面写入shellcode，然后在子程序直接执行。问题是里面存在沙箱保护。所以我们得先看一下沙箱里ban了什么。

可以看出来，沙箱很奇怪，orw也没给我们留，只给了一个叫ptrace的函数。

那我们就得了解一下什么是ptrace。

ptrace，即process trace，从其名字上就能看出，ptrace()系统调用能提供追踪进程执行状态的功能。根据官方手册的介绍：ptrace()系统调用为一个进程提供了观察和控制另一个进程的执行过程的能力，同时也提供检查和改变另一个进程的内存值以及相关注册信息。其中，被控制的进程被称为tracee，控制进程被称为tracer。

ptrace的函数签名如下：

  • request：要执行的操作类型；
  • pid：被追踪的目标进程ID；
  • addr：被监控的目标内存地址；
  • data：保存读取出或者要写入的数据。
参数 含义
request 请求类型
pid 被调试进程的 PID
addr 地址参数（依赖于请求类型）
data 数据参数（依赖于请求类型）

常见的请求宏

宏名 对应的数值 描述
PTRACE_TRACEME 0 表示自己要被父进程调试（子进程调用）
PTRACE_PEEKDATA / PTRACE_PEEKUSER 2/ 3 读取数据或用户区（寄存器）
PTRACE_POKEDATA/ PTRACE_POKEUSER 5 / 6 写入数据或用户区（寄存器）
PTRACE_GETREGS / PTRACE_SETREGS 12/ 13 获取或设置所有通用寄存器（x86、x86_64）
PTRACE_CONT 7 继续执行被暂停的子进程
PTRACE_SINGLESTEP 9 单步执行
PTRACE_ATTACH 16 附加到一个正在运行的进程（类似 GDB attach）
PTRACE_DETACH 17 从目标进程分离，目标继续运行
PTRACE_SYSCALL 24 每次系统调用前暂停目标，常用于 syscall hook
PTRACE_KILL 8 杀死被调试进程
PTRACE_SEIZE 0x4206 (16902) 非侵入式 attach，用于新型调试接口
PTRACE_INTERRUPT 0x4207 (16903) 中断目标（常用于 SEIZE 模式）

在寄存器结构(user_regs_struct)中，rip存放在偏移为0x80处，所以rip的地址应该是0xDEADC0DE580，之后我们去把rsp移到0xDEADC0DE588的位置，又把装有我们恶意指令的rax给push进栈。这个时候rsp-8，也就是说现在0xDEADC0DE580即rip里存放着的地址就是我们的恶意指令的起始位置。然后最后又把rsp给移到恶意指令的起始。

4.canutrytry

还是给了一个菜单函数。

这里漏洞点在于选项3处。这里没有去检测v16的大小。所以我们完全可以通过输入一个合适的负值，来修改stdout的结构体。

加上程序已经把flag读入到了s_0可控地址（pie未开）。所以我们利用stdout就可以把flag直接输出了。但是由于我们在写入的时候还需要注意size[v16]>0。所以如果我们按照我们算出来的chunks[v16]来做的话，sizes[v16]就刚好是0。所以我们需要找一个合适的偏移来保证能修改到stdout的同时，至少size>0。

我们这个时候再来看leave函数。

leave函数这里有个典型的漏洞。栈溢出。可以看出是先进行memcpy才去检查size是不是>16。然后会触发异常处理。

那这里就要触及到什么是c++的异常处理了。

核心原理：异常展开（Unwinding）

当 leave() 函数中满足条件抛出异常时（__cxa_throw）：

  1. 程序不会继续往下执行，也不会执行该函数的 Epilogue（即不会执行 Canary 校验和 ret 指令）。
  2. 相反，它会调用底层的 _Unwind_RaiseException 开始栈展开。
  3. 栈展开器（Unwinder）会根据栈上的返回地址（Return Address）去查找 .eh_frame 段，找到调用者（比如 main 函数）的异常处理代码（catch 块）。
  4. 最关键的一步：展开器会从当前溢出后的栈上，读取调用者原本保存的寄存器状态(特别是 RBP），并将寄存器恢复，然后直接 jmp 到 main 的 catch 块。

利用思路： 我们故意溢出覆盖 Canary 和 Saved RBP，但绝对不覆盖返回地址。这样异常展开器能正常找到 main 函数，但它会将我们伪造的 RBP 恢复给 main。当 main 结束并执行 leave; ret 时，就会顺理成章地将栈劫持到我们控制的地址！

如果那个从 0x401EEA 开始的 try 块抛出了类型为 char const*（字符串指针）的异常，就跳到 0x401F7B 这里来处理。

我们在 leave() 函数里抛出的刚好是 throw "stack overflow";，正是一个常量字符串指针！然后会跳转到0x401F8D去处理。

关键可以看最后一段。会把libc基地址和栈地址打印出来。拿到libc基地址，我们也可以确定stdout的地址了。然后就是修改stdout来打印出flag。
